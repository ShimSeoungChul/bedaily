## 개요

명세와 구현체. 같은 무대를 두 이름으로 부르는 두 층을 나란히 놓고, 어디가 표준이고 어디가 Hibernate가 얹은 것인지 구분해 읽는 글이다. 기준은 **Spring Data JPA 3 / Hibernate 6** — JPA 4 / Hibernate 7은 별도로 다룬다.

아래 다이어그램에서 색은 두 층을 가른다.

- **청록(teal)** — JPA: 명세 / 표준 (Jakarta Persistence)
- **앰버(amber)** — Hibernate: 구현체 (ORM Provider)

- **JPA** *(명세)* — 명세일 뿐 동작 코드가 없다. 인터페이스와 애너테이션의 규약만 정의한다. 단독으로는 아무것도 못 돌린다.
- **Hibernate** *(구현체)* — JPA 명세를 구현한 가장 널리 쓰이는 provider. 표준 인터페이스를 제공하면서 표준을 넘어서는 고유 기능도 함께 가진다.
- **Spring Data JPA** — JPA를 대체하지 않고 그 위에 얹는 추상화. `JpaRepository.save()`는 내부에서 상태를 보고 표준 메서드를 고르는, 아래 Hibernate 메서드와는 전혀 다른 물건이다.

표를 읽기 전에, 이 둘이 **무슨 일을 하는 물건인지**부터 — 아래 속성은 전부 이 역할에서 따라 나온다.

- **Factory** *(EntityManagerFactory / SessionFactory)* — 부팅 시 1번 생성돼 앱 종료까지 사는 단일 객체. 매핑 메타데이터·쿼리 플랜·2차 캐시 같은 **공유 상태**를 보유하고, 요청마다 `EntityManager`를 찍어낸다 → 그래서 **무겁고 thread-safe**.
- **Manager** *(EntityManager / Session)* — `persist`·`find`·`merge`의 실제 진입점. 트랜잭션 단위로 **영속성 컨텍스트**(가변 상태)를 쌓는다 → 그래서 **가볍고 thread-unsafe**.

표준이 정한 이름과 Hibernate가 구현한 이름은 이렇게 짝지어진다.

- **EntityManagerFactory** *(JPA)* ↔ **SessionFactory** *(Hibernate)* — 위 **Factory** 역할. 부팅 시 한 번 만들어 앱당 하나로 산다.
- **EntityManager** *(JPA)* ↔ **Session** *(Hibernate)* — 위 **Manager** 역할. 요청/트랜잭션 단위로 영속성 컨텍스트를 들고 있다.

> **짚어둘 점** — **영속성 컨텍스트는 JPA 표준 개념이다.** Hibernate 고유 구조가 아니다. 1차 캐시·변경 감지·쓰기 지연·엔티티 생명주기가 전부 JPA 명세에 들어 있다. Hibernate는 이 무대를 `Session` 내부의 `PersistenceContext`로 *구현* 했을 뿐이다. Hibernate 7부터 `SessionFactory.createEntityManager()`가 편의상 `Session`을 반환하는 것도 Session이 EntityManager의 상위 호환이라는 신호다.

- **Transient** — `new`로 막 만든 객체. 식별자도 없고 컨텍스트가 모른다. `persist()`를 호출해야 Managed로 넘어간다.
- **Managed** — `persist()` 또는 `find()`/JPQL 조회로 컨텍스트에 올라온 상태. 필드를 바꾸면 dirty checking이 잡아 flush 시 UPDATE가 자동으로 나간다. 별도 save 호출이 필요 없는 핵심 상태.
- **Detached** — 트랜잭션 종료, `clear()`, `close()`, `detach()`로 컨텍스트에서 분리된 상태. 필드를 바꿔도 DB에 반영되지 않는다. 다시 Managed로 되돌리려면 `merge()`를 호출한다 — 원본이 아닌 *새 managed 복사본*이 반환되므로, 이후 작업은 반환된 객체로 해야 한다.
- **Removed** — `remove()` 호출로 삭제 예정 표시. 실제 DELETE는 flush 시점에 나간다. 아직 flush 전이면 `persist()`로 다시 Managed로 되돌릴 수 있다.

> **표준 vs 고유 — 재부착의 차이** — Detached를 다시 다루는 방식이 갈린다. Hibernate 고유 `update()`는 *넘긴 객체 자체* 를 managed로 되돌린다(재부착) — 같은 PK의 managed 인스턴스가 이미 있으면 `NonUniqueObjectException`. JPA 표준 `merge()`는 상태를 복사한 *새 managed 객체* 를 반환하고 원본은 detached로 남겨, 이 충돌 자체가 생기지 않는다. 그래서 표준은 merge 하나로 정리됐다.

### flush를 일으키는 세 가지

- **트랜잭션 commit** — 가장 확실한 트리거. 커밋 직전 항상 flush가 일어난다.
- **JPQL/Criteria 쿼리 실행** — `AUTO` 모드 기본 동작. 쿼리가 방금 변경한 데이터를 못 보면 결과가 틀어지므로, 쿼리 전에 관련 변경분을 먼저 내보낸다. **네이티브 SQL은 이 자동 flush 보장이 약하므로 주의.**
- **명시적 flush()** — `em.flush()` 직접 호출. 배치에서 메모리 관리나 중간 ID 확보가 필요할 때.

### FlushModeType — 자동 flush 범위 조절

```java filename="FlushMode.java"
// AUTO (기본) — 커밋 + 쿼리 실행 시 flush
// COMMIT      — 커밋 시에만 flush, 쿼리 실행으로는 안 함
em.setFlushMode(FlushModeType.COMMIT);

// 흔한 함정: persist 후 같은 트랜잭션에서 JPQL 조회
em.persist(member);                           // 버퍼에만
List<Member> list = em.createQuery(           // AUTO면 여기서 자동 flush →
    "select m from Member m", Member.class)   // 방금 persist한 member도 보인다
    .getResultList();
// 반면 find(id)는 1차 캐시에서 바로 꺼내므로 flush와 무관
```

> **실무 주의** — flush가 트랜잭션을 커밋하는 게 아니다. flush 후에도 롤백하면 DB는 원복된다. 또 `FlushModeType.COMMIT`으로 바꾸면 쿼리 직전 자동 flush가 사라져, 방금 변경한 데이터가 같은 트랜잭션 쿼리에 안 잡힐 수 있다 — 성능 최적화로 쓰되 정합성 영향을 반드시 검토할 것.

### 5가지 전략

- **IDENTITY** *(JPA 표준)* — DB의 auto_increment / IDENTITY 컬럼에 위임. MySQL `AUTO_INCREMENT`, PostgreSQL `SERIAL`. INSERT를 해야 PK가 나오므로 **persist 즉시 INSERT가 나가고 JDBC 배치가 막힌다.**
- **SEQUENCE** *(JPA 표준)* — DB 시퀀스 객체에서 INSERT 전에 값을 받아온다. Oracle·PostgreSQL 등. 쓰기 지연과 배치 INSERT가 그대로 살아 있어 대량 쓰기에 유리. `allocationSize`로 블록 선할당 가능.
- **TABLE** *(JPA 표준)* — 별도 키 관리 테이블로 시퀀스를 흉내. 모든 DB에서 동작하지만 테이블 조회·락 오버헤드로 가장 느리다. 사실상 비권장.
- **AUTO** *(JPA 표준)* — JPA 기본값. provider가 dialect를 보고 결정 — Hibernate 6에선 시퀀스 지원 DB면 SEQUENCE를 고른다. 운영에선 명시적 지정이 더 예측 가능하다.
- **UUID** *(Hibernate)* — Hibernate 6부터 `@UuidGenerator`로 표준화. DB 왕복 없이 애플리케이션에서 PK 생성 — 분산 환경에 적합. 단 랜덤 UUID의 인덱스 단편화는 별도 고려(정렬 가능한 UUIDv7 등).

### 전략별 비교 — ID 확보 시점이 모든 걸 가른다

| 전략 | ID 확보 시점 | persist 시 SQL | 배치 INSERT |
|---|---|---|---|
| IDENTITY *(JPA)* | INSERT 직후 (DB가 매김) | 즉시 INSERT | 불가 |
| SEQUENCE *(JPA)* | INSERT 전 (시퀀스 조회) | 버퍼링 (지연) | 가능 |
| TABLE *(JPA)* | INSERT 전 (테이블 조회) | 버퍼링 (지연) | 가능 (느림) |
| UUID *(Hibernate)* | persist 시 앱에서 생성 | 버퍼링 (지연) | 가능 |

### allocationSize — SEQUENCE 성능의 핵심

Hibernate 6은 엔티티별 시퀀스를 기본으로 만들고, JPA 표준에 따라 `allocationSize` 기본값이 50이다(JPA 명세상 2.0부터 쭉 50 — Hibernate 4 이전의 hilo 옵티마이저가 사실상 1처럼 동작했던 것과 혼동하지 말 것). 이 값이 DB 왕복 횟수를 직접 좌우한다.

- **allocationSize = 1** — 풀링 없음. ID가 필요할 때마다 매번 `nextval()` 호출 → INSERT마다 DB 왕복 1회.
- **allocationSize = 50 (기본)** — pooled 옵티마이저 동작. 시퀀스를 한 번 당겨 50개 ID를 메모리에서 소진 → DB 왕복이 1/50로 감소.

```java filename="Member.java"
// 엔티티별 시퀀스 + 블록 선할당 (Hibernate 6 기본 방향)
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "member_seq")
@SequenceGenerator(name = "member_seq", sequenceName = "member_seq",
                   allocationSize = 50)
private Long id;
```

> **주의 — 옵티마이저 정합** — pooled 옵티마이저는 DB 시퀀스의 increment가 엔티티 `allocationSize`와 같아야 정상 동작한다. 어긋나면 ID 갭이나 충돌이 생긴다. Hibernate가 DDL을 생성하면 자동으로 맞지만, 시퀀스를 수동 생성했거나 `import.sql`로 초기 데이터를 넣을 때는 직접 맞춰야 한다.

> **앞 절과의 연결** — 이것이 flush 원칙의 예외 지점이다. SEQUENCE·TABLE·UUID는 ID를 INSERT 전에 확보하므로 persist가 SQL을 버퍼링하지만, IDENTITY만은 ID 회수를 위해 persist 즉시 INSERT를 던진다. 대량 INSERT가 잦은 테이블이라면 이 차이가 곧 성능 차이다.

## Fetch 전략과 N+1

연관 엔티티를 언제 로딩하느냐의 문제. 기본 전략을 잘못 잡거나 LAZY를 무심코 순회하면 쿼리가 폭발한다. JPA 실무에서 가장 자주 마주치는 성능 함정.

### LAZY vs EAGER

- **LAZY (지연)** — 연관 엔티티를 프록시로 두고, 실제 접근하는 순간 쿼리. `@OneToMany`·`@ManyToMany`의 기본값.
- **EAGER (즉시)** — 엔티티 조회 시 연관까지 함께 로딩. `@ManyToOne`·`@OneToOne`의 기본값. **실무에선 모두 LAZY로 두고 필요할 때 fetch join을 권장.**

### LAZY의 내부 구현 — 프록시와 컬렉션 래퍼

"실제 접근하는 순간 쿼리"는 어떻게 가능한가? Hibernate는 **프록시 패턴**으로 이를 구현한다. 엔티티 단건(`@ManyToOne`)에는 **ByteBuddy로 생성한 프록시 서브클래스**를, 컬렉션(`@OneToMany`)에는 **PersistentCollection 래퍼**를 끼운다. 두 경우 모두 "접근 시점에 가로채서 초기화"라는 같은 원리다.

#### 단건 연관 — 프록시 서브클래스

`@ManyToOne(fetch = LAZY)`로 선언된 필드를 로딩하면 Hibernate는 실제 엔티티가 아니라 **해당 클래스를 상속한 프록시 객체**를 채워 넣는다. 이 프록시는 PK만 들고 있고, 나머지 필드는 비어 있다. getter를 호출하는 순간 인터셉터가 작동해 DB를 조회하고 실제 값을 채운다.

```java filename="LazyProxy.java"
// getName()은 실제 데이터 접근 → 이 시점에 SELECT 발생
team.getName();                         // SELECT * FROM team WHERE id=?

// 컬렉션도 마찬가지 — 접근 시점에 초기화
team.getMembers().size();               // SELECT * FROM member WHERE team_id=?

// 초기화 여부 확인 유틸
Hibernate.isInitialized(team);          // true/false
Hibernate.initialize(team);            // 명시적 강제 초기화
```

> **왜 이 구조를 알아야 하나** — LAZY가 프록시라는 걸 모르면 세 가지 함정에 빠진다. ① `instanceof` 검사가 프록시 타입에 걸려 실패, ② 트랜잭션 밖에서 프록시 접근 시 `LazyInitializationException`, ③ 프록시를 루프에서 건드리면 N+1. 아래 N+1 절은 ③의 구체적 사례다.

#### OSIV — 컨트롤러에서 LAZY가 되는 이유

"트랜잭션이 서비스에서 끝났는데 왜 컨트롤러에서 LAZY 접근이 되지?" — Spring Boot는 **OSIV(Open Session In View)**가 기본 활성화(`spring.jpa.open-in-view=true`)되어 있기 때문이다. OSIV는 HTTP 요청의 시작부터 응답까지 Session을 열어두므로 컨트롤러·뷰에서도 프록시 초기화가 가능하다.

편리하지만 대가가 있다. Session이 요청 전체에 걸쳐 살아 있으면 **DB 커넥션도 그만큼 오래 잡힌다**. 트래픽이 늘면 커넥션 풀 고갈로 이어진다. 또 어디서든 LAZY 접근이 되니 N+1이 뷰 레이어까지 퍼져도 눈에 안 띈다.

```yaml filename="application.yml"
# 권장 — OSIV 끄기
spring:
  jpa:
    open-in-view: false   # 서비스 계층 밖에서 프록시 접근 시 즉시 예외
                           # → LAZY 접근 지점을 서비스 안에서 명시적으로 관리하게 된다
```

> **실무 권장** — OSIV를 끄고, 필요한 연관은 서비스 계층에서 fetch join·EntityGraph·DTO 프로젝션으로 미리 해결한다. "어디서든 접근 가능"보다 "어디서 접근하는지 명확한" 코드가 더 안전하다.

```java filename="NPlusOne.java"
// ❌ N+1 발생 — members가 LAZY일 때 루프에서 초기화
List<Team> teams = em.createQuery("select t from Team t", Team.class)
                       .getResultList();           // 쿼리 1번
for (Team t : teams) {
    t.getMembers().size();                          // 팀마다 쿼리 1번씩 → N번
}

// ✅ fetch join — 한 방에
List<Team> teams = em.createQuery(
    "select distinct t from Team t join fetch t.members", Team.class)
    .getResultList();                               // 쿼리 1번으로 끝
```

### 해법 정리

- **fetch join** — JPQL `join fetch`로 연관을 한 쿼리에 적재. 가장 직접적. 단, 컬렉션 fetch join은 페이징과 충돌(메모리 페이징 경고)하니 주의.
- **@BatchSize** *(Hibernate)* — LAZY 로딩을 `IN (?, ?, …)`로 묶어 N번을 1번으로. 전역 설정은 `default_batch_fetch_size`.
- **@EntityGraph** *(JPA 표준)* — 어떤 연관을 함께 EAGER로 가져올지 그래프로 선언. Spring Data 메서드에 바로 얹기 좋다.

```java filename="EntityGraphExample.java"
// ✅ @EntityGraph — Spring Data 메서드에 바로 얹는 JPA 표준 방식
public interface TeamRepository extends JpaRepository<Team, Long> {

    @EntityGraph(attributePaths = {"members"})   // members를 함께 EAGER로 적재
    List<Team> findAll();                        // 쿼리 1번 — 내부적으로 LEFT JOIN

    @EntityGraph(attributePaths = {"members", "members.department"})
    Optional<Team> findById(Long id);            // 2단계 연관까지 한 번에
}

// ✅ @BatchSize — 전역 설정으로 모든 LAZY 컬렉션에 일괄 적용
// application.yml:
//   spring.jpa.properties.hibernate.default_batch_fetch_size: 100
//
// 또는 엔티티별로 직접 지정:
@Entity
public class Team {
    @BatchSize(size = 100)                       // 이 컬렉션만 IN 절로 묶음
    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members;
}
// → 팀 N개 조회 후 members 접근 시: WHERE team_id IN (1,2,3,...,100) 1번으로 해결
```

> **권장 기본기** — 모든 연관을 `LAZY`로 깔아 의도치 않은 EAGER 폭발을 막고, 필요한 화면·쿼리에서만 fetch join이나 EntityGraph로 명시적으로 끌어온다. "기본은 안 가져옴, 필요할 때만 함께"가 안전한 출발점이다.

## 2차 캐시 — 표준이 끝나는 지점

### 2차 캐시란 무엇인가

앞 절에서 본 1차 캐시(영속성 컨텍스트)는 **트랜잭션 하나**의 생애와 함께한다. 트랜잭션이 끝나면 그 안에 쌓인 캐시도 전부 사라진다. 같은 엔티티를 다른 트랜잭션에서 다시 조회하면 — 아무리 1초 전에 읽었어도 — DB로 SELECT가 나간다.

2차 캐시(L2 Cache)는 이 한계를 넘는다. **애플리케이션 전체(SessionFactory)가 공유하는 캐시**다. 한 트랜잭션이 읽은 엔티티를 2차 캐시에 올려두면, 다른 트랜잭션이 같은 PK를 조회할 때 DB를 치지 않고 캐시에서 꺼낸다. 변경이 드물고 읽기가 잦은 데이터 — 코드 테이블, 카테고리, 설정 값 같은 — 에 효과가 크다.

### 표준과 구현의 경계

JPA는 2차 캐시의 **활성화 규약**만 정한다. 실제로 데이터를 어디에 어떻게 저장할지는 전부 provider 영역이다.

- **JPA가 정하는 것** — `@Cacheable` 애너테이션, `shared-cache-mode`(어떤 엔티티를 캐시할지), `Cache` API(수동 evict). 여기까지가 표준.
- **provider가 정하는 것** *(Hibernate)* — 캐시 저장소(Ehcache·Caffeine·Infinispan), 만료 정책(TTL·max-size), 동시성 전략(`READ_ONLY`·`READ_WRITE`·`NONSTRICT_READ_WRITE`·`TRANSACTIONAL`). 이 설정들은 전부 Hibernate 고유 애너테이션이나 프로퍼티로 들어간다.

### 내부 조회 흐름 — L1 → L2 → DB

`em.find(Member.class, 1L)`을 호출하면 Hibernate는 세 단계를 순서대로 탐색한다. 2차 캐시에 저장되는 것은 **엔티티 객체가 아니라 분해된 상태값(dehydrated state)** — `{id=1, name="Lee", age=28}` 같은 배열 형태다. 꺼낼 때 이 값으로 새 엔티티 인스턴스를 조립(hydrate)해서 1차 캐시에 넣는다.

> **왜 엔티티 객체를 직접 저장하지 않는가** — 2차 캐시에 managed 엔티티를 그대로 넣으면 여러 트랜잭션이 같은 인스턴스를 공유하게 된다. 한쪽이 필드를 바꾸면 다른 쪽도 영향받아 dirty checking이 깨지고, 트랜잭션 격리가 무너진다. 그래서 Hibernate는 **분해된 스칼라 배열(dehydrated state)**로 저장하고, 꺼낼 때마다 새 인스턴스를 조립한다.

### 설정과 코드 예제

2차 캐시를 켜려면 세 가지가 필요하다: ① 캐시 라이브러리 의존성, ② 프로퍼티 활성화, ③ 엔티티에 캐시 애너테이션.

```java filename="Category.java"
// ③ 엔티티에 캐시 선언
@Entity
@Cacheable                                       // JPA 표준 — "이 엔티티는 2차 캐시 대상"
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE,
       region = "categoryCache")                 // Hibernate 고유 — 동시성 전략·리전 지정
public class Category {

    @Id @GeneratedValue(strategy = GenerationType.SEQUENCE)
    private Long id;

    private String name;

    @Cache(usage = CacheConcurrencyStrategy.READ_WRITE)  // 컬렉션도 별도로 캐시 선언
    @OneToMany(mappedBy = "category", fetch = FetchType.LAZY)
    private List<Product> products = new ArrayList<>();
}
```

```yaml filename="application.yml"
# ① 의존성: org.hibernate.orm:hibernate-jcache + org.ehcache:ehcache (또는 Caffeine)
# ② 프로퍼티 활성화
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true        # 2차 캐시 켜기
          use_query_cache: true               # 쿼리 결과 캐시 (선택)
          region.factory_class: jcache        # Hibernate 6 — JCache 표준 팩토리
      jakarta:
        persistence:
```

#### 현실적 선택지

| 방식 | 동작 | 장단점 |
|---|---|---|
| **로컬 캐시 + 짧은 TTL** | Caffeine·Ehcache로 TTL 30초~5분 | 간단하지만 TTL 동안 불일치 허용 필요 |
| **분산 캐시 (Redis·Infinispan)** | 모든 인스턴스가 하나의 캐시 공유 | 정합성 해결, 대신 네트워크 홉 추가 |
| **Pub/Sub 무효화** | 변경 시 Redis Pub/Sub·Kafka로 evict 전파 | 정합성 좋지만 인프라 복잡도 증가 |
| **2차 캐시 안 쓰기** | Spring Cache + Redis로 서비스 레이어에서 직접 | 가장 많은 팀이 택하는 방향 |

```java filename="ServiceLayerCache.java"
// 현대적 대안 — JPA 2차 캐시 대신 서비스 레이어에서 Spring Cache 사용
@Service
public class CategoryService {

    @Cacheable(value = "categories", key = "#id")   // Spring Cache → Redis
    public CategoryDto findById(Long id) {
        return categoryRepository.findById(id)
            .map(CategoryDto::from)
            .orElseThrow();
    }

    @CacheEvict(value = "categories", key = "#id")  // 변경 시 명시적 무효화
    public void update(Long id, CategoryUpdateRequest req) {
        Category cat = categoryRepository.findById(id).orElseThrow();
        cat.update(req);
    }
}
```

> **현실적 판단** — JPA 2차 캐시가 빛나는 장면은 여전히 있다: **단일 인스턴스**, 또는 **변경이 거의 없는 코드성 데이터**(국가 코드, 권한 테이블, 카테고리 트리). 이런 데이터는 `READ_ONLY` 전략으로 올려두면 DB 부하를 눈에 띄게 줄인다. 그러나 멀티 인스턴스에서 변경이 잦은 비즈니스 데이터까지 JPA 2차 캐시로 커버하려 하면 — 정합성 관리 비용이 성능 이득을 빠르게 넘어선다. 많은 팀이 **JPA 2차 캐시는 끄고, 서비스 레이어에서 Spring Cache(@Cacheable) + Redis**로 캐시 경계를 직접 제어하는 방향을 택하는 이유다.

- **JPQL** *(JPA)* — 테이블이 아니라 엔티티를 대상으로 하는 표준 쿼리 언어. `select m from Member m where m.age > :age`.
- **Criteria API** *(JPA)* — 타입 안전한 동적 쿼리를 자바 코드로. 표준이지만 장황해서 실무 선호도는 낮은 편.
- **Native Query** *(JPA)* — SQL 직접 작성. DB 종속 기능이 필요할 때. Hibernate 쿼리 엔진을 거치지 않으므로 자동 flush 보장이 약한 점은 flush 절 참고.
- **@NamedQuery** *(JPA)* — 이름 붙인 정적 쿼리. 부팅 시 문법 검증되는 이점.
- **QueryDSL** *(서드파티)* — JPA 표준이 아닌 서드파티. 타입 안전한 동적 쿼리를 Criteria보다 읽기 쉽게. 내부적으로는 JPQL로 번역돼 결국 표준 위에서 돈다.

## 메서드 대응표 — 표준 vs 고유

Hibernate가 JPA보다 먼저 나왔기에 고유 메서드가 먼저 있었고, 나중에 JPA 표준 메서드를 추가로 얹었다. 두 계열이 오래 공존했고, **Hibernate 6에서는 고유 계열이 deprecated**로 표시돼 있다(동작은 그대로, 경고만). 실제 제거는 Hibernate 7부터다.

| Hibernate 고유 (6: deprecated) | JPA 표준 (권장 대체) | 핵심 차이 |
|---|---|---|
| `session.save()` | `persist()` | save는 ID 즉시 반환 위해 INSERT를 앞당기는 경향, persist는 flush까지 지연 |
| `session.update()` | `merge()` | update는 원본을 재부착(PK 충돌 위험), merge는 복사본 반환 |
| `session.saveOrUpdate()` | `persist() / merge()` | 신규·기존 자동 분기 → 표준은 상황에 맞게 둘 중 선택 |
| `session.delete()` | `remove()` | 명칭 통일 |
| `session.get()` | `find()` | 즉시 SELECT, 없으면 null — 동작 동일, 표준 명칭으로 |
| `session.load()` | `getReference()` | DB 조회 없이 프록시(참조)만 — 가장 실용적이던 용도, 그대로 계승 |

대체 매핑(save→persist, update→merge, get→find, load→getReference)과 실제 "제거"는 Hibernate 7 기준이며 별도로 다룬다. 위 표는 6 기준 deprecated 상태로 읽을 것.

### 왜 표준은 merge 하나로 갔나

```java filename="MergeVsUpdate.java"
// Hibernate update() — 넘긴 객체 그 자체가 managed로 (재부착)
session.update(detached);   // 같은 PK managed 있으면 NonUniqueObjectException

// JPA merge() — 상태를 복사한 새 managed 객체를 반환
Member managed = em.merge(detached);  // 원본은 detached 유지, 충돌 자체가 없음
```

> **Spring Data JPA의 save는 다른 물건** — `JpaRepository.save()`는 위 `Session.save()`와 이름만 같다. Spring Data가 엔티티 상태(새 것인지 여부)를 보고 내부에서 `persist` 또는 `merge`를 고르는 추상화다. 그래서 나중에 Hibernate 7로 올라가 고유 메서드가 제거돼도 repository 코드는 영향받지 않는다 — 영향받는 건 `EntityManager.unwrap(Session.class)`로 Session을 직접 꺼내 고유 메서드를 부르는 레거시 코드뿐이다.
