## 개요

Elasticsearch는 "Apache Lucene 위에 세운 분산 검색·분석 엔진"이다. JSON 문서를 색인하면 단어 기반 전문 검색과 조건별 통계 처리를 빠르게 수행한다.

검색, 로그·관측성, 분석 대시보드 같은 워크로드에 적합하다. 트랜잭션·조인 중심의 주 저장소가 아니라 보조 계층이라는 점을 먼저 짚고 가자. Elasticsearch의 설계 철학은 수평 확장, 고가용성, near real-time이다.

- **Index**: 매핑과 세팅을 가진 논리 단위다. ES7부터 타입이 하나(`_doc`)로 통합됐다.
- **Shard**: 인덱스의 조각이자 독립된 Lucene 인덱스. ES7의 기본값은 primary 1, replica 1이다 (ES6의 primary 5에서 변경).
- **Segment**: 불변 파일 묶음이다. 생성 후 수정할 수 없고, 삭제는 표시만 해둔다.

## 샤드 라우팅 · 복제 · 장애 복구

### 라우팅 공식

문서가 어느 샤드에 저장될지는 다음 공식으로 결정된다.

```
shard = hash(_routing) % primary_shard_count
```

`_routing`의 기본값은 문서 `_id`다. 이 공식이 Primary 샤드 수가 생성 후 변경 불가인 이유다. 샤드 수가 바뀌면 동일 `_id`의 라우팅 결과가 달라져서 문서를 찾을 수 없게 된다.

### 읽기

읽기는 Primary/Replica 아무 사본에서 가능하다. ES7은 기본적으로 **adaptive replica selection**을 사용해서 응답이 빠른 사본을 우선 선택한다.

### 복제 내부

- **seq_no · primary_term**: 쓰기 연산마다 시퀀스 번호와 primary 임기를 부여한다. 장애 시 어느 연산까지 복제됐는지 추적할 수 있다.
- **체크포인트 기반 복구**: local/global checkpoint으로 복제 상태를 추적하고, 노드 복구 시 마지막 체크포인트 이후만 재전송한다.

핵심은 **refresh ≠ flush**라는 것이다. refresh는 검색 가능 상태를 만들고(기본 1초), flush는 디스크에 영구 저장한다. 이 1초 간격이 Elasticsearch가 "near real-time"인 이유다. 색인 직후 바로 검색되지 않는다.

```yaml
# 급한 경우: 색인 후 즉시 검색 가능하게
POST /my-index/_doc/1?refresh=wait_for

# 처리량 우선: 대량 적재 시 refresh 끄기
index.refresh_interval: -1
```

**Translog**은 내구성을 담당한다. flush 전에 노드가 죽으면 translog를 재생해서 데이터를 복구한다.

```yaml
# 기본값: 매 요청마다 fsync (안전하지만 느림)
index.translog.durability: request

# 성능 우선: 비동기 sync (유실 위험 있음)
index.translog.durability: async
index.translog.sync_interval: 5s
```

**Merge**는 세그먼트 관리의 핵심이다. 세그먼트는 불변이므로 update는 "기존 문서 삭제 표시 + 새 문서 추가"로 처리된다. 삭제 표시된 문서는 merge 때 실제로 정리된다. 색인이 완료된 인덱스에 `_forcemerge`를 실행하면 검색 성능이 올라간다.

Query 단계에서는 모든 샤드에 검색을 보내고, 각 샤드는 `_id`와 점수만 돌려준다. Coordinating 노드가 전역 정렬 후 상위 N건을 확정하면, Fetch 단계에서 해당 샤드에서만 본문(`_source`)을 회수한다.

**Deep paging 함정**: `from: 10000, size: 10`이면 각 샤드가 10,010건을 정렬해서 보내야 한다. 해결책은 `search_after`, `scroll`, 또는 7.10+ Point in Time(PIT)을 사용하는 것이다.

**집계(Aggregation)**: 검색과 별개 경로로, 각 샤드가 부분 집계 후 coordinating이 합산한다. `doc_values`(디스크 컬럼 저장)를 사용해서 고속화한다.

**스코어링**: ES7 기본은 **BM25** (TF-IDF 개선판)이다. `filter` 컨텍스트는 점수 계산을 건너뛰고 캐시된다.

### 샤드 사이징

- 샤드 1개당 **10~50GB**를 권장한다.
- 노드당 샤드 수는 힙 1GB당 20개 이하가 경험칙이다.
- `cluster.max_shards_per_node` 기본값은 1000.
- ES7 기본이 primary 1이라 over-sharding은 줄었지만, 시계열 인덱스가 많으면 여전히 주의해야 한다.

### 인덱싱 성능

```yaml
# 기본: Bulk API로 묶음 전송 (단건 X)
# 배치 크기: 5~15MB부터 튜닝 시작

# 대량 적재 시 최적화
index.refresh_interval: -1
index.number_of_replicas: 0
# 완료 후 원복 + _forcemerge 실행
```

### 안정성·운영

| 항목 | 설정 | 비고 |
|------|------|------|
| Master 노드 | 3개 (홀수) | 투표 기반 합의 |
| 디스크 워터마크 | low 85% / high 90% / flood 95% | flood에서 쓰기 차단 |
| Circuit breaker | `use_real_memory: true` | 실제 힙 사용량 기반 |
| 존 분산 | `allocation.awareness` | 노드를 AZ별로 분산 |
| 보안 | TLS + 기본 인증 + RBAC | 기본 라이선스에 포함 |

### 백업·복구

- **스냅샷**: `_snapshot` API로 S3·공유 파일시스템에 백업. 증분 방식으로 변경 세그먼트만 저장한다.
- **Searchable snapshots** (7.12+): 스냅샷을 frozen 티어로 마운트해서 검색 가능한 상태로 유지한다.
- **샤드 재배치**: 노드 추가/제거 시 자동 재분배된다.

### 인덱스 관리

- **Index template**: 새 인덱스에 매핑·세팅 자동 적용. 7.8+의 Component template으로 조합 가능.
- **Alias**: 애플리케이션은 항상 alias로 접근한다. 무중단 인덱스 전환이 가능하다.
- **매핑 주의**: 필드 타입은 변경 불가(추가만 가능). `dynamic: strict` 또는 명시적 매핑을 권장한다.

## 마무리

Elasticsearch 7은 **Index → Shard → Segment**로 내려가는 계층 구조 위에서 동작한다. 이 글을 관통하는 핵심 세 가지를 기억하자.

1. **"샤드는 분산의 단위이자 Lucene 인덱스 그 자체다"** — 샤드 수 = Lucene 인스턴스 수. 너무 많으면 오버헤드, 너무 적으면 확장 한계.
2. **"Refresh와 flush는 다른 일이다"** — refresh는 검색 가능, flush는 디스크 영구 저장. 이 구분이 near real-time과 내구성을 모두 설명한다.
3. **"Primary 샤드 수는 라우팅에 묶여 생성 후 변경 불가"** — 운영 중 변경 가능한 건 replica 수뿐이다. 초기 설계가 중요한 이유.
